## SequenceCodec 설계 의도 : 왜 문자열 대신 정수 배열을 사용하는가?

본 프로젝트의 `SequenceCodec`은 입력된 단백질/유전체 서열의 문자(A, R, N, D...)를 0~20 사이의 1바이트 정수(`uint8_t`) 또는 표준 정수(`int`) 배열로 사전 인코딩합니다. 단순한 문자열 비교 로직을 배제하고 이러한 변환 과정을 거치는 이유는 **시스템 연산 성능**을 끌어내기 위함이며, 그 세 가지 핵심 이유는 다음과 같습니다.

### 1. 분기문 제거, $O(1)$ 직접 참조

단순 문자열 기반 매칭은 필연적으로 내부 조건문(`if (seq1 == seq2)`)을 요구합니다. 이러한 조건 분기(Branch)는 CPU의 Pipeline Stall을 유발하고, 특히 GPU 환경에서는 코어 간의 동기화를 깨뜨리는 **워프 발산**의 주범이 됩니다.
서열을 정수로 인코딩해 두면, 조건문 없이 BLOSUM62와 같은 2차원 치환 행렬의 인덱스로 즉시 매핑하여 `score_matrix[seq1[i]][seq2[j]]` 형태로 $O(1)$ 시간에 점수를 도출할 수 있습니다.

### 2. Memory Density와 Cache Hit 활용

아미노산을 아스키코드(ASCII) 그대로 치환 행렬의 인덱스로 사용하려면, 영문자를 모두 덮을 수 있는 최소 `128 x 128` (16,384칸) 크기의 메모리 공간이 필요하며 이 중 90% 이상은 사용되지 않는 낭비 공간이 됩니다.

하지만 20개의 아미노산을 0~19의 연속된 정수로 압축(Encoding)하면, 점수 행렬을 `20 x 20` (400칸)이라는 매우 작고 조밀한 크기로 다이어트할 수 있습니다. 이 크기는 CPU의 L1 캐시나 GPU의 초고속 공유 메모리(Shared Memory)에 완벽하게 탑재되어, 메인 메모리(RAM) 병목을 원천 차단합니다.

### 3. PCIe 대역폭 및 하드웨어 친화성

프로세서의 연산 장치(ALU)는 Text보다 Numeric 처리에 최적화되어 있습니다. 서열을 1바이트 정수(`uint8_t`) 배열로 변환함으로써 메모리 정렬의 이점을 얻을 뿐만 아니라, 향후 대용량 서열 데이터를 GPU로 복사할 때 **PCIe 버스의 Bandwidth 오버헤드를 절약**할 수 있습니다.

## Troubleshooting : GPU 병목 분석 및 공유 타일링 적용

### 개요 (Expectation vs. Reality)
스미스-워터맨 알고리즘의 점수 테이블 크기가 약 430만 칸(2,100 x 2,074)에 달했을 때, 수천 개의 코어를 가진 GPU가 단일 코어 기반의 CPU를 압도할 것으로 예측했습니다.

>CPU의 시간 복잡도: $O(N \times M)$단일 코어 CPU는 이 400만 개의 칸을 2중 for문을 돌며 한 번에 한 칸씩 400만 번 계산해야 합니다.
>GPU의 시간 복잡도: $O(N + M)$대각선 위에 있는 칸들은 서로 의존성이 없으므로 동시에 계산할 수 있습니다. 테이블을 가로지르는 대각선의 총개수는 $N + M - 1$ 개입니다. (약 4,000개)GPU는 수천 개의 코어를 동원해 대각선 1줄을 한번에 계산 할 수 있습니다.

그러나 1차 글로벌 메모리 웨이브프론트 최적화(`sw_gpu.cu`) 테스트 결과, **CPU가 238ms를 기록한 반면 GPU는 373ms를 기록하며 참패**하는 기현상이 발생했습니다. 하드웨어 스펙을 역행하는 이 병목 현상의 3가지 치명적 원인은 다음과 같습니다.

---

### 1. 글로벌 메모리 병합 실패 (Uncoalesced Memory Access) - [가장 치명적인 원인]
GPU 메모리 아키텍처의 가장 중요한 원칙인 '메모리 병합(Coalescing)'이 완전히 파괴되었기 때문입니다.

* **CPU의 캐시 친화성:** CPU는 2차원 배열을 가로 방향(행)으로 순차 탐색하므로, 메모리 주소가 연속적으로 읽혀 L1/L2 캐시 적중률(Hit Rate)이 극도로 높습니다.
* **GPU의 메모리 트랜잭션 낭비 (파편화):** 대각선(Wavefront) 방향으로 연산하는 GPU 스레드들을 1차원 배열 인덱스(Flattened Array) 관점에서 뜯어보면 심각한 병목이 드러납니다.
    * Thread 0: `(i, j)` 연산 → 메모리 주소: `i * cols + j`
    * Thread 1: `(i-1, j+1)` 연산 → 메모리 주소: `(i-1) * cols + j + 1`
* **결과:** 워프(Warp) 내에서 바로 옆에 있는 스레드들이 서로 약 2,000칸(`cols - 1`)씩 떨어진 엉뚱한 메모리 주소를 동시에 요구하게 됩니다. GPU 하드웨어는 단 4 Byte의 정수 하나를 읽기 위해서라도 최소 128 Byte(Cache Line) 단위로 데이터를 통째로 가져와야만 합니다. 스레드들이 이처럼 뿔뿔이 흩어진 주소를 요청한 탓에, 하드웨어는 불필요한 128 Byte 덩어리들을 강제로 수십만 번 긁어오게 되었습니다.
* 결국 실제 연산에 필요한 데이터보다 수십 배 많은 쓰레기 데이터가 메모리 대역폭(Bandwidth)을 꽉 채워버리며 코어가 데이터를 기다리느라 멈춰버리는 극심한 메모리 대기(Stall) 현상을 유발했습니다.

### 2. 폭발적인 커널 런칭 오버헤드 (Kernel Launch Overhead)
GPU 연산은 Host(CPU)가 Device(GPU)에게 "이 코드를 실행해라"라고 명령을 내리는 구조입니다. 이 명령(`<<<...>>>`)을 내리고 동기화하는 데는 필연적으로 마이크로초($\mu s$) 단위의 지연 시간이 발생합니다.

* 대각선 웨이브프론트 방식에서는 **대각선 1줄을 계산할 때마다 새로운 커널을 런칭**해야 합니다.
* 2,000 x 2,000 크기의 테이블에서는 총 $N + M - 1$, 즉 **약 4,171번의 커널 호출**이 발생했습니다. 
* GPU가 실제 연산을 수행하는 시간보다, CPU의 명령을 기다리고 껐다 켜지는(Context Switching) 오버헤드가 누적되어 배보다 배꼽이 커지는 현상이 발생했습니다.

### 3. GPU 코어 굶주림 (Starvation)
현대적인 GPU는 수십 개의 SM(Streaming Multiprocessor)과 수천 개의 CUDA 코어를 가지고 있어, 한 번에 수만 개의 스레드를 쏟아부어야 제 성능을 발휘합니다.

* 테이블의 양 끝단(시작과 끝부분) 대각선은 계산해야 할 칸 수가 1개, 2개, 3개 식으로 매우 적습니다.
* 가장 긴 대각선(가운데)조차 최대 길이가 2,074칸(약 8~9개 블록)에 불과합니다.
* **결과:** GPU 입장에서는 일거리가 너무 적어서, 전체 코어의 10%도 채 쓰지 못하고 대부분의 코어가 전원만 켜둔 채 굶고 있는(Starvation) 상태가 지속되었습니다.

---

### 💡 결론 및 타개책
글로벌 메모리에 직접 접근하는 단순 웨이브프론트 병렬화는 **메모리 I/O 지연이 연산 속도를 갉아먹는 전형적인 Memory-Bound 문제**를 야기합니다.

이를 해결하기 위해 전체 테이블을 $32 \times 32$ 단위로 쪼개어, 연속된 메모리를 한 번에 긁어와 초고속 캐시인 **공유 메모리(Shared Memory)**에 적재한 뒤 연산하는 **Shared Memory Tiling** 아키텍처(`sw_gpu_tiled.cu`)를 도입해야만 진정한 하드웨어 가속 성능을 끌어낼 수 있습니다.
