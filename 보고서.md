# Smith-Waterman 서열 정렬 알고리즘의 CUDA 병렬화 및 성능 평가

---

## 1. 서론

### 가. 선택 알고리즘(기술) 소개

#### 1) 생물학적 배경: 서열 정렬(Sequence Alignment)이란

생명정보학(Bioinformatics)에서 **서열 정렬(Sequence Alignment)** 은 두 개 이상의 생물학적 서열(DNA의 염기서열 A·T·G·C, 또는 단백질의 아미노산 서열 20종)을 나란히 배치하여, 진화적·기능적으로 유사한 영역을 찾아내는 핵심 기법이다. 두 단백질이 공통 조상으로부터 유래했는지, 특정 유전자가 어떤 기능을 수행하는지, 변이(mutation)가 어디에서 발생했는지를 추론하는 모든 분석은 결국 "두 서열이 얼마나 닮았는가"를 정량화하는 정렬 문제로 귀결된다.

서열 정렬은 크게 두 가지로 나뉜다.

- **전역 정렬(Global Alignment, Needleman-Wunsch):** 두 서열을 처음부터 끝까지 전체를 대상으로 정렬한다. 길이가 비슷하고 전반적으로 유사한 서열에 적합하다.
- **지역 정렬(Local Alignment, Smith-Waterman):** 두 서열 중 **부분적으로 가장 잘 일치하는 구간(substring)** 만을 찾아 정렬한다. 길이 차이가 크거나, 전체적으로는 다르지만 특정 도메인(domain)만 보존된 서열을 비교할 때 사용한다.

본 프로젝트가 선택한 알고리즘은 후자인 **스미스-워터맨(Smith-Waterman, 이하 SW)** 알고리즘이다.

#### 2) 가) Smith-Waterman 알고리즘의 정의와 특징

SW 알고리즘은 1981년 Temple Smith와 Michael Waterman이 제안한 **동적 계획법(Dynamic Programming, DP)** 기반의 지역 정렬 알고리즘이다. Needleman-Wunsch 전역 정렬 알고리즘을 변형하여, 점수가 음수가 되면 0으로 잘라내는(clipping) 방식으로 "지역적으로 가장 강한 유사 구간"을 추출한다.

SW가 학술적·산업적으로 중요한 이유는 **최적해(optimal solution)를 수학적으로 보장**하기 때문이다. BLAST나 FASTA 같은 휴리스틱(heuristic) 도구는 빠르지만 근사해를 내놓는 반면, SW는 주어진 점수 모델 하에서 반드시 최고 점수의 정렬을 찾아낸다. 따라서 정확도가 생명을 좌우하는 분야(신약 후보 단백질 검색, 임상 변이 분석 등)에서 황금 표준(gold standard)으로 쓰인다.

#### 3) (1) SW의 점수 모델: BLOSUM62 치환 행렬과 갭 페널티

SW의 정렬 품질은 **점수 모델(scoring scheme)** 에 의해 결정된다. 본 구현은 단백질 서열을 대상으로 하므로 다음 두 요소를 사용한다.

- **치환 행렬(Substitution Matrix) — BLOSUM62:** 두 아미노산이 정렬될 때 부여하는 점수표이다. 단순히 "같으면 +1, 다르면 -1"이 아니라, **생화학적으로 유사한 아미노산끼리는 치환되어도 진화적으로 자주 관찰되므로 높은 점수**를 준다. 예를 들어 류신(L)과 이소류신(I)은 둘 다 소수성(hydrophobic)이라 치환되어도 +2점이지만, 류신(L)과 프롤린(P)은 구조적으로 매우 달라 -3점이다. BLOSUM62는 62% 이상 동일한 단백질 블록들로부터 통계적으로 유도한 20×20 행렬로, 단백질 정렬의 사실상 표준이다. 본 구현의 `common.h`에 20×20 정수 배열로 하드코딩되어 있다.
- **갭 페널티(Gap Penalty):** 한쪽 서열에 삽입(insertion)/삭제(deletion)가 일어나 빈칸('-')을 넣어야 할 때 부여하는 벌점이다. 본 구현은 갭 하나당 일정한 벌점을 부여하는 **선형 갭(linear gap)** 모델을 사용하며, `common.h`에 `GAP_PENALTY = -2`로 정의되어 있다.

### 나. 병렬화 선택 이유 및 기대 효과

#### 1) 왜 병렬화가 필요한가 — 문제의 규모

SW의 시간 복잡도는 두 서열 길이 m, n에 대해 **O(mn)** 이다. 짧은 서열(수백 AA)에서는 무시할 수 있지만, 본 프로젝트가 벤치마크로 삼은 **티틴(Titin)** 단백질은 인간 단백질 중 가장 큰 약 **34,350 AA**, 생쥐 티틴은 **35,213 AA** 에 달한다. 두 서열을 정렬하면 점수 테이블의 크기만

> 35,214 × 34,351 ≈ **약 12억(1.21 × 10⁹) 셀**, `int`(4 byte) 기준 **약 4.5 GB**

에 이른다. CPU 단일 코어로 12억 개 셀을 순차 계산하면 본 실험 환경에서 **약 67초(66,973 ms)** 가 걸린다. 데이터베이스 검색처럼 수만 개 서열과 반복 비교해야 하는 실제 워크로드에서는 이 비용이 곧바로 병목이 된다.

#### 2) 왜 GPU(CUDA)인가 — 데이터 병렬성의 적합성

SW의 핵심 연산은 "각 셀마다 이웃 세 셀을 보고 최댓값을 취하는" 단순하고 동일한 산술 연산이 12억 번 반복되는 구조다. 이는 **수천 개의 코어가 동일한 연산을 서로 다른 데이터에 적용하는 SIMT(Single Instruction, Multiple Threads)** 모델, 즉 GPU에 이상적으로 들어맞는다.

다만 SW에는 한 가지 걸림돌이 있다. 각 셀 H(i,j)는 **왼쪽·위·왼쪽위 세 셀에 의존**하므로 모든 셀을 동시에 계산할 수는 없다. 본 프로젝트는 이 데이터 의존성을 **반대각선(anti-diagonal) 독립성** 으로 우회한다. 같은 반대각선(i+j=k) 위의 셀들은 서로 의존하지 않으므로 한꺼번에 병렬 계산할 수 있다(상세는 2장·3장).

#### 3) 기대 효과

- **연산 스텝 수의 점근적 개선:** 순차 알고리즘은 셀을 하나씩 처리하므로 O(mn) 스텝이지만, 반대각선 단위로 묶으면 전체 대각선 개수만큼만 순차 진행하면 되어 **O(m+n) 스텝**으로 줄어든다. 각 대각선 내부는 완전 병렬이다.
- **메모리 계층 최적화를 통한 처리량 증대:** 단순 GPU 이식만으로는 느린 전역 메모리(Global Memory) 접근이 병목이 된다. 본 프로젝트는 **공유 메모리 타일링(Shared Memory Tiling)** 과 **uint8_t 서열 인코딩**, **상수 메모리(Constant Memory) 치환 행렬 배치**를 통해 메모리 대역폭 효율을 끌어올린다.
- **정량적 목표:** CPU 베이스라인 대비 의미 있는 가속(speedup)을 달성하고, 세 구현(CPU / 기본 GPU / 타일 GPU)이 **동일한 정렬 점수**를 산출하여 병렬화의 정확성을 검증한다.

## 2. 순차 알고리즘 분석

### 가. 알고리즘 설명 및 의사 코드

#### 1) 점화식(Recurrence Relation)

SW는 두 서열 A(길이 m), B(길이 n)에 대해 (m+1)×(n+1) 크기의 점수 테이블 H를 채운다. 0번 행과 0번 열은 모두 0으로 초기화하며, 나머지 셀은 다음 점화식으로 계산한다.

```
H(i, j) = max {
    0,                                  // (1) 새로운 지역 정렬의 시작 (음수 차단)
    H(i-1, j-1) + s(A[i], B[j]),        // (2) 대각선: 매치 / 미스매치
    H(i-1, j)   + gap,                  // (3) 위쪽:   A에 대한 갭 (B의 삽입)
    H(i,   j-1) + gap                   // (4) 왼쪽:   B에 대한 갭 (A의 삽입)
}
```

여기서 `s(A[i], B[j])`는 BLOSUM62 치환 점수, `gap`은 갭 페널티(-2)이다. 네 항 중 **첫 번째 항 `0`** 이 SW를 전역 정렬과 구분 짓는 핵심이다. 점수가 음수로 떨어지면 0으로 리셋하여, "지금까지의 정렬을 버리고 이 위치에서 새 지역 정렬을 시작"하는 효과를 낸다.

#### 2) 두 단계: 테이블 채우기 + 트레이스백

SW는 두 단계로 이루어진다.

1. **테이블 채우기(Fill):** 위 점화식으로 전체 테이블을 채우면서 **최댓값 셀의 위치(max_i, max_j)와 점수(max_score)** 를 기록한다. 전역 정렬과 달리 최적 정렬의 끝점은 테이블의 모서리가 아니라 점수가 가장 높은 임의의 셀이다.
2. **트레이스백(Traceback):** 최댓값 셀에서 출발해, 각 셀의 점수가 세 이웃 중 어디에서 유래했는지 역추적하며 **점수가 0이 되는 셀에 도달할 때까지** 경로를 따라간다. 이 경로가 최적 지역 정렬이다. 대각선 이동은 매치/미스매치, 위·왼쪽 이동은 갭('-')에 대응한다.

#### 3) 의사 코드

```
function SmithWaterman(A[1..m], B[1..n]):
    # --- 1단계: 테이블 채우기 ---
    allocate H[0..m][0..n], 모든 셀 0으로 초기화
    max_score, max_i, max_j ← 0, 0, 0

    for i ← 1 to m:
        for j ← 1 to n:
            diag ← H[i-1][j-1] + s(A[i], B[j])
            up   ← H[i-1][j]   + GAP_PENALTY
            left ← H[i][j-1]   + GAP_PENALTY
            H[i][j] ← max(0, diag, up, left)

            if H[i][j] > max_score:
                max_score ← H[i][j]
                (max_i, max_j) ← (i, j)

    # --- 2단계: 트레이스백 ---
    (i, j) ← (max_i, max_j)
    align_A, align_B ← "", ""
    while i > 0 and j > 0 and H[i][j] > 0:
        if H[i][j] == H[i-1][j-1] + s(A[i], B[j]):   # 대각선
            앞에 A[i], B[j] 추가;  i--; j--
        else if H[i][j] == H[i-1][j] + GAP:          # 위
            앞에 A[i], '-' 추가;  i--
        else:                                        # 왼쪽
            앞에 '-', B[j] 추가;  j--
    return (max_score, align_A, align_B)
```

#### 4) 실제 구현 대응

위 의사 코드는 본 프로젝트의 [sw_cpu.cpp](1_sw_implement/sw_cpu.cpp#L61-L108) `smith_waterman_cpu()` 함수와 [common.h](1_sw_implement/common.h#L128-L190) `run_traceback()` 함수에 그대로 대응된다. 한 가지 구현상의 최적화는, 치환 점수 조회 시 **분기문 없이 배열을 직접 참조**한다는 점이다.

```cpp
// sw_cpu.cpp:89-90 — 분기 없이 2차원 배열 직접 참조, O(1)
int match_mis_score =
    (idx1 < 20 && idx2 < 20) ? BLOSUM62[idx1][idx2] : -4;
```

서열 문자를 미리 0~19의 정수로 인코딩(SequenceCodec)해 두었기에, `if (A==B)` 같은 문자 비교 없이 `BLOSUM62[idx1][idx2]`로 즉시 점수를 가져온다. 'X' 등 미지 문자(인덱스 20)는 -4점으로 처리한다.

### 나. 시간/공간 복잡도

| 구분 | 복잡도 | 설명 |
|---|---|---|
| **시간(테이블 채우기)** | **O(m·n)** | (m+1)×(n+1) 셀 각각을 상수 시간 O(1)에 계산 |
| **시간(트레이스백)** | O(m+n) | 경로 길이는 최대 m+n에 비례 |
| **공간(전체 테이블)** | **O(m·n)** | 트레이스백을 위해 전체 테이블 보관 필수 |

티틴 벤치마크(35,213 × 34,350) 기준 시간은 약 12억 회의 셀 연산, 공간은 약 4.5 GB이다.

> **참고 — 공간 복잡도와 트레이스백의 트레이드오프:** 만약 최종 점수(max_score)만 필요하고 정렬 경로가 필요 없다면, 점화식이 직전 두 대각선(또는 직전 한 행)에만 의존하므로 공간을 **O(min(m,n))** 으로 줄일 수 있다. 그러나 트레이스백으로 실제 정렬 문자열을 복원하려면 전체 테이블이 있어야 하므로, 본 구현은 O(mn) 공간을 사용한다. 이 4.5 GB 테이블이 GPU 메모리 대역폭 병목의 근본 원인이 되며, 3장·6장의 최적화 논의의 출발점이 된다.

### 다. 병렬화 가능성 분석

#### 1) 데이터 의존성(Data Dependency)

순차 알고리즘을 그대로 병렬화할 수 없는 이유는 점화식의 의존 구조 때문이다. 셀 H(i,j)는

- H(i-1, j-1) (왼쪽 위)
- H(i-1, j) (위)
- H(i, j-1) (왼쪽)

세 셀이 **모두 먼저 계산되어 있어야** 한다. 즉 같은 행(row)을 좌→우로 채울 때, H(i,j)는 바로 왼쪽 H(i,j-1)에 의존하므로 한 행 내부의 셀들조차 동시에 계산할 수 없다. 행 단위·열 단위 병렬화는 불가능하다.

#### 2) 반대각선 독립성(Anti-diagonal Independence) — 웨이브프론트(Wavefront)

핵심 통찰은 **같은 반대각선 위의 셀들은 서로 의존하지 않는다**는 점이다. 반대각선이란 `i + j = k`(상수)를 만족하는 셀들의 집합이다. 셀 H(i,j)가 의존하는 세 이웃을 보면,

- H(i-1, j) 와 H(i, j-1) → `i+j = k-1` (직전 반대각선)
- H(i-1, j-1) → `i+j = k-2` (전전 반대각선)

즉 **k번째 반대각선의 모든 셀은 k-1, k-2 반대각선에만 의존하고, 같은 k번째 반대각선 내부의 셀끼리는 서로 독립적**이다. 따라서 k=2부터 k=m+n까지 반대각선을 하나씩 순차적으로 진행하되, **각 반대각선 내부의 모든 셀은 동시에 병렬 계산**할 수 있다. 마치 파도(wave)가 좌상단에서 우하단으로 대각선 모양으로 전진하는 형태라 하여 이를 **웨이브프론트(Wavefront) 병렬화** 라 부른다.

```
   B →   b1  b2  b3  b4
A   ┌────────────────────
↓ a1│   k2  k3  k4  k5      같은 숫자(k) = 같은 반대각선
  a2│   k3  k4  k5  k6      → 동시 계산 가능
  a3│   k4  k5  k6  k7
  a4│   k5  k6  k7  k8
```

#### 3) 병렬화의 효과와 한계

- **효과:** 순차적으로 진행해야 하는 스텝이 셀 단위(O(mn))에서 반대각선 단위(**O(m+n)**)로 감소한다. 티틴의 경우 약 12억 스텝이 약 7만 스텝(35,213+34,350-1 = 69,562개 대각선)으로 줄어든다.
- **한계 1 — 부하 불균형:** 반대각선의 길이는 일정하지 않다. 좌상단·우하단 꼭짓점 근처에서는 셀이 1개뿐이지만, 중앙 대각선은 최대 min(m,n)개에 달한다. 짧은 대각선을 처리할 때는 대부분의 GPU 코어가 유휴(idle) 상태가 되어 점유율(occupancy)이 떨어진다.
- **한계 2 — 동기화 비용:** 각 반대각선은 직전 대각선이 **완전히 끝나야** 시작할 수 있으므로, 대각선마다 전역 동기화(barrier)가 필요하다. 이 동기화를 어떻게 구현하느냐가 GPU 성능을 좌우한다(3장에서 두 가지 설계를 비교).

## 3. CUDA 병렬화 설계

### 가. 병렬화 전략

본 프로젝트는 웨이브프론트 병렬화를 **두 단계로 발전**시켰다. 두 버전을 모두 구현하여 메모리 최적화의 효과를 정량적으로 비교한다.

#### 1) 1차 설계 — 셀 단위 웨이브프론트 ([sw_gpu.cu](1_sw_implement/sw_gpu.cu))

가장 직관적인 매핑이다. **하나의 GPU 스레드가 하나의 셀**을 담당한다. 반대각선 동기화는 **커널 런치(kernel launch)의 경계** 를 이용한다. 즉, `for` 루프를 돌며 반대각선마다 커널을 한 번씩 호출한다. 호스트(CPU)에서 다음 커널을 호출하는 순간, 이전 커널(이전 대각선)은 이미 완료가 보장되므로 별도의 전역 동기화 코드가 필요 없다.

```cpp
// sw_gpu.cu:133-144 — 대각선 개수(69,562회)만큼 커널을 반복 런치
for (int d = 1; d <= num_diagonals; d++) {
    int num_elements_in_diag = ...;
    int blocksPerGrid = (num_elements_in_diag + threadsPerBlock - 1) / threadsPerBlock;
    sw_kernel_wavefront<<<blocksPerGrid, threadsPerBlock>>>(...);
}
```

- **장점:** 구현이 단순하고 명확하다.
- **단점:** 티틴 기준 **커널을 69,562회** 런치한다. 커널 런치마다 수 µs의 오버헤드가 발생하고, 각 셀이 의존 셀 3개를 매번 **느린 전역 메모리에서 다시 읽어온다.** 데이터 재사용(data reuse)이 전혀 없어 메모리 대역폭에 묶인(memory-bound) 상태가 된다.

#### 2) 2차 설계 — 타일 단위 매크로 웨이브프론트 ([sw_gpu_tiled.cu](1_sw_implement/sw_gpu_tiled.cu))

1차 설계의 두 병목(잦은 커널 런치 + 전역 메모리 재접근)을 **타일링(Tiling)** 으로 해결한다. 전체 테이블을 **32×32 크기의 타일**로 나누고, **하나의 스레드 블록이 하나의 타일 전체**를 담당한다.

웨이브프론트를 **두 계층**으로 구성한다.

- **매크로 웨이브프론트(타일 레벨):** 타일들도 반대각선 의존 구조를 따른다. 타일 대각선(block diagonal) 단위로 커널을 런치한다. 티틴 기준 타일 행 1,101개 × 타일 열 1,074개 → 타일 대각선은 약 **2,172개**. 즉 **커널 런치가 69,562회 → 약 2,172회로 32배 감소**한다.
- **마이크로 웨이브프론트(타일 내부):** 한 블록(32 스레드)이 타일 내부의 32×32 셀을 **공유 메모리 위에서** 반대각선 순서(2×32-1 = 63개 내부 대각선)로 계산한다. 내부 대각선마다 `__syncthreads()`로 블록 내 동기화한다.

```cpp
// sw_gpu_tiled.cu:119-140 — 타일 내부 웨이브프론트 (공유 메모리에서 수행)
for (int d = 0; d < 2 * TILE_SIZE - 1; d++) {
    int i = min(d, TILE_SIZE - 1) - tx + 1;
    int j = max(0, d - TILE_SIZE + 1) + tx + 1;
    if (유효 범위) {
        int diag = H_s[i-1][j-1] + match_mis_score;
        int up   = H_s[i-1][j]   + GAP_PENALTY;
        int left = H_s[i][j-1]   + GAP_PENALTY;
        H_s[i][j] = max(0, max(diag, max(up, left)));
    }
    __syncthreads();   // 블록 내 동기화 (커널 재런치보다 훨씬 저렴)
}
```

타일 내부의 63번의 동기화는 전역 메모리를 거치지 않고 **블록 내부 동기화(`__syncthreads`)** 로 처리되며, 32×32 셀의 모든 중간 계산이 **공유 메모리(`H_s`)** 에서 이루어지므로 전역 메모리 트래픽이 급감한다.

### 나. 메모리 계층 활용

GPU 성능은 결국 **어떤 데이터를 어느 메모리에 두느냐**로 결정된다. CUDA 메모리 계층은 속도순으로 레지스터 > 공유 메모리/L1 > L2 캐시 > 전역 메모리(DRAM) 이며, 본 프로젝트는 데이터 특성에 맞게 세 종류의 메모리를 전략적으로 사용한다.

#### 1) 상수 메모리(Constant Memory) — BLOSUM62 치환 행렬

BLOSUM62(20×20 = 400개 정수)는 **모든 스레드가 읽기만 하고, 절대 변하지 않는** 데이터다. 이는 상수 메모리의 이상적 사용 사례다. 상수 메모리는 전용 캐시를 가지며, **워프 내 모든 스레드가 같은 주소를 읽을 때 한 번의 읽기로 브로드캐스트(broadcast)** 되어 매우 효율적이다.

```cpp
// sw_gpu.cu:14, sw_gpu_tiled.cu:17
__constant__ int d_BLOSUM62[20][20];
// ... cudaMemcpyToSymbol(d_BLOSUM62, BLOSUM62, 20*20*sizeof(int));
```

#### 2) 공유 메모리(Shared Memory) — 타일 점수 테이블 (타일 버전의 핵심)

타일 버전은 32×32 셀의 작업 영역과 의존 경계, 두 서열 조각을 모두 공유 메모리로 끌어올린다.

```cpp
// sw_gpu_tiled.cu:83-85
__shared__ uint8_t seq1_s[TILE_SIZE];          //  타일이 담당하는 서열1 조각 32개
__shared__ uint8_t seq2_s[TILE_SIZE];          //  서열2 조각 32개
__shared__ int     H_s[TILE_SIZE + 1][TILE_SIZE + 1];  //  타일 점수 + 위/왼쪽 경계
```

처리 흐름은 **"전역 → 공유 메모리로 일괄 적재 → 공유 메모리에서 63회 반복 연산 → 결과를 전역으로 일괄 기록"** 이다. 타일 내부의 모든 중간 읽기/쓰기가 전역 메모리(수백 cycle 지연) 대신 공유 메모리(수 cycle 지연)에서 일어나므로, 전역 메모리 접근은 **타일당 경계 적재 1회 + 결과 기록 1회**로 최소화된다. 1차 설계가 셀마다 전역 메모리를 3번 읽던 것과 대조된다.

특히 의존 경계 처리에 주의가 필요하다. 타일의 결과는 인접한 다음 타일의 입력(위쪽·왼쪽 경계)이 되므로, 커널 시작 시 전역 테이블에서 위쪽 행(`H_s[0][*]`)·왼쪽 열(`H_s[*][0]`)·좌상단 모서리(`H_s[0][0]`)를 공유 메모리로 읽어와 의존성을 잇는다.

```cpp
// sw_gpu_tiled.cu:99-113 — 타일 간 의존성을 잇는 경계 로드
if (start_i > 1 && ...) H_s[0][tx+1]   = score_table[(start_i-1)*cols + (start_j+tx)]; // 위 경계
if (start_j > 1 && ...) H_s[tx+1][0]   = score_table[(start_i+tx)*cols + (start_j-1)]; // 왼 경계
if (tx == 0)            H_s[0][0]       = score_table[(start_i-1)*cols + (start_j-1)];  // 좌상단
```

#### 3) 전역 메모리(Global Memory) — 4.5 GB 점수 테이블과 코얼레싱(Coalescing)

전체 4.5 GB 점수 테이블은 GPU L2 캐시(수 MB)에 절대 담기지 않으므로 모든 접근이 DRAM에서 일어난다. 따라서 **연속된 스레드가 연속된 메모리 주소에 접근하는 코얼레싱(memory coalescing)** 패턴이 성능에 결정적이다. 타일 버전은 결과를 다시 쓸 때 각 스레드가 한 행을 담당하여, 워프 단위로 인접 주소에 접근하도록 설계해 메모리 트랜잭션을 합친다.

```cpp
// sw_gpu_tiled.cu:144-150 — 결과를 전역 테이블에 일괄 기록 (coalesced write)
if (start_i + tx <= len1) {
    for (int curr_j = 1; curr_j <= TILE_SIZE; curr_j++)
        if (start_j + curr_j - 1 <= len2)
            score_table[(start_i + tx) * cols + (start_j + curr_j - 1)] = H_s[tx + 1][curr_j];
}
```

#### 4) uint8_t 서열 인코딩 — 데이터 밀도(Data Density) 4배

서열 자체도 메모리 최적화 대상이다. 본 프로젝트의 `SequenceCodec`([sequence_codec.h](1_sw_implement/sequence_codec.h))는 아미노산 문자열 `"ARNDCQEGHILKMFPSTWYV"` 를 0~19의 **1바이트 정수(`uint8_t`)** 로 인코딩한다. 표준 `int`(4 byte) 대신 `uint8_t`(1 byte)를 쓰면 동일한 서열이 **4배 높은 밀도**로 메모리에 적재되어, 한 번의 캐시 라인(cache line) 읽기로 4배 많은 문자를 가져온다.

이 효과는 Nsight Compute로 실측 검증되었다([99_archive/profiling](99_archive/profiling/)). 동일 데이터에 대해 `int` 버전이 `uint8_t` 버전보다 **DRAM 읽기량이 약 4.6배** 많았다. 이는 (1) PCIe로 GPU에 서열을 복사할 때의 대역폭 절약, (2) 캐시 라인 활용률(Cache Line Utilization) 향상, (3) 워프 발산(warp divergence) 없는 O(1) 점수 조회를 동시에 달성한다.

### 다. 커널 설계

#### 1) 셀 단위 커널 `sw_kernel_wavefront` (1차)

각 스레드가 자신의 전역 인덱스(tid)로부터 담당할 셀 좌표 (i, j)를 역산한다. 반대각선 d의 시작 좌표에서 tid만큼 떨어진 셀을 맡는다.

```cpp
// sw_gpu.cu:64-99 (요약)
int tid = blockIdx.x * blockDim.x + threadIdx.x;
int start_i = min(d, len1);
int start_j = max(1, d - len1 + 1);
int i = start_i - tid;        // 반대각선을 따라 좌표 분해
int j = start_j + tid;
if (테이블 범위 내) {
    int diag = score_table[(i-1)*cols + (j-1)] + match_mis_score; // 전역 메모리 읽기
    int up   = score_table[(i-1)*cols +  j   ] + GAP_PENALTY;
    int left = score_table[ i   *cols + (j-1)] + GAP_PENALTY;
    score_table[i*cols + j] = max(0, max(diag, max(up, left)));   // 전역 메모리 쓰기
}
```

- **그리드 구성:** 블록당 256 스레드. 그리드 크기는 해당 대각선의 셀 개수에 맞춰 동적 할당하여 불필요한 스레드를 최소화한다.
- **clipping to 0:** `max(0, ...)`로 음수를 차단하여 SW의 지역 정렬 의미를 구현한다.

#### 2) 타일 커널 `sw_kernel_tiled` (2차)

블록 인덱스로부터 타일의 2D 좌표(block_i, block_j)를 역산한다. 타일 대각선 인덱스가 정사각 테이블의 절반을 넘었는지에 따라 좌표 공식이 갈린다(대각선이 길어지다가 다시 짧아지는 구조 반영).

```cpp
// sw_gpu_tiled.cu:70-78
if (block_diag_idx < num_blocks_row) {        // 전반부: 대각선이 길어지는 구간
    block_i = block_diag_idx - b;  block_j = b;
} else {                                       // 후반부: 대각선이 짧아지는 구간
    block_i = (num_blocks_row - 1) - b;
    block_j = (block_diag_idx - num_blocks_row + 1) + b;
}
```

커널 본체는 ① 공유 메모리 적재 → ② `__syncthreads()` → ③ 내부 63 대각선 웨이브프론트 → ④ 전역 기록 의 4단계로 구성된다(앞서 인용한 코드). **스레드 수를 타일 폭(32)과 동일하게 매핑**하여, 32 스레드가 워프 하나에 깔끔히 들어맞도록 했다(워프 = 32 스레드).

#### 3) 정확성 보장 설계

세 구현 모두 동일한 점화식·치환 행렬·갭 페널티를 사용하며, 경계 조건(테이블 밖 접근, 미지 문자 'X' 처리)을 일관되게 처리한다. 타일 버전은 테이블 끝의 자투리 타일에서 범위를 벗어나는 셀을 'X'(인덱스 20)로 채우거나 0으로 처리하여 정확성을 유지한다(`sw_gpu_tiled.cu:92-96, 124-136`). 그 결과 세 구현이 **동일한 최대 점수(165,624)와 정렬 길이(35,318 AA)** 를 산출함이 실험에서 확인되었다(5장).

실제 단백질 데이터는 FASTA 포맷(헤더 `>`로 시작, 서열이 여러 줄로 분할)으로 배포된다. C++17 `<filesystem>` 기반 전처리기가 `input_fasta/`의 모든 `.fasta`/`.fa` 파일을 순회하며, 헤더와 줄바꿈을 제거해 순수 서열만 추출하여 `output_sequence/`에 `*_seq1.txt` 형태로 저장한다. 이로써 정렬 단계의 입출력을 단순화하고(Decoupled Preprocessing), 정렬 커널이 순수 연산에만 집중하도록 했다.

### 나. 핵심 코드 설명

#### 1) 인코딩 — 분기 없는 O(1) 매핑 ([sequence_codec.h](1_sw_implement/sequence_codec.h#L15-L47))

생성자에서 128칸 ASCII 룩업 테이블을 만들어, 각 아미노산 문자를 즉시 인덱스로 변환한다. 미지 문자는 기본값 20('X')으로 매핑되어 별도 분기 없이 처리된다.

```cpp
SequenceCodec() {
    for (int i = 0; i < 128; i++) ascii_2_index[i] = 20;   // 기본값 'X'
    string amino_acids = "ARNDCQEGHILKMFPSTWYV";
    for (int i = 0; i < 20; i++) ascii_2_index[amino_acids[i]] = i; // A→0, R→1, ...
}
vector<uint8_t> encode(string &sequence) {
    vector<uint8_t> out(sequence.length());
    for (int i = 0; i < sequence.length(); i++)
        out[i] = ascii_2_index[sequence[i]];   // 룩업 한 번, 분기 없음
    return out;
}
```

#### 2) GPU 파이프라인 — 측정 범위와 메모리 전송 ([sw_gpu_tiled.cu](1_sw_implement/sw_gpu_tiled.cu#L25-L62))

`main`은 ① `cudaFree(0)`로 CUDA 컨텍스트를 미리 초기화(warmup)하여 측정 밖으로 빼고, ② 타이머를 켠 뒤 전체 GPU 파이프라인(H2D 복사 + 커널 + D2H 복사)을 실행하고, ③ 타이머를 끈다. 즉 **순수 커널 시간이 아니라 데이터 전송을 포함한 실사용 시간**을 측정하여 공정성을 확보한다.

```cpp
cudaFree(0);   // 컨텍스트 초기화 패널티를 타이머 밖으로 배제
auto start = high_resolution_clock::now();
SWResult result = smith_waterman_gpu(seq1_int, seq2_int);  // H2D + Kernel + D2H 전부 포함
auto end   = high_resolution_clock::now();
```

#### 3) 견고한 오류 처리 — `CUDA_CHECK` 매크로 ([common.h](1_sw_implement/common.h#L19-L30))

모든 CUDA API 호출을 매크로로 감싸 즉시 오류를 검출한다. 4.5 GB `cudaMalloc`처럼 실패 가능성이 있는 호출에서 오류를 조용히 삼키지 않고 파일·라인과 함께 보고한 뒤 종료한다(최근 커밋 `7baa24e`에서 "CUDA 오류 무시 버그"를 이 매크로 도입으로 수정).

```cpp
#define CUDA_CHECK(call) do {                                  \
    cudaError_t _err = (call);                                 \
    if (_err != cudaSuccess) {                                 \
        std::cerr << "[CUDA ERROR] " << cudaGetErrorString(_err) \
                  << " (" << __FILE__ << ":" << __LINE__ << ")"; \
        exit(EXIT_FAILURE);                                    \
    }                                                          \
} while (0)
```

#### 4) 트레이스백 ([common.h](1_sw_implement/common.h#L128-L190))

최댓값 셀에서 출발해 점수의 유래(대각선/위/왼쪽)를 역추적한다. 점수가 0이 되면 지역 정렬의 시작점이므로 종료한다. 정렬 결과 문자열을 뒤집어 출력한다. 이 단계는 본질적으로 순차적이며 CPU에서 수행한다.

### 다. 구현시 고려 사항

1. **측정 공정성:** GPU의 첫 호출에는 컨텍스트 초기화 비용(수백 ms)이 섞이므로 `cudaFree(0)` 워밍업으로 분리했다. 또한 GPU 측정에 **데이터 전송(H2D/D2H)을 포함**시켜, "커널만 빠르고 실제로는 느린" 착시를 배제했다.
2. **수치 정확성 우선:** 점수 테이블을 `int`로 유지했다. 티틴의 최대 점수가 165,624로 16비트(±32,767) 범위를 초과하므로 `int16`/`uint8` 누산은 오버플로를 일으킨다. 메모리를 더 쓰더라도 정확성을 택했다.
3. **자투리 타일(boundary tile) 처리:** 서열 길이가 32의 배수가 아니므로 테이블 가장자리에 불완전한 타일이 생긴다. 범위를 벗어나는 셀을 'X'/0으로 안전하게 채워 잘못된 메모리 접근과 점수 오염을 방지했다.
4. **워프 정렬:** 타일 폭(32)을 워프 크기(32)와 일치시켜, 한 타일의 한 내부 대각선을 정확히 한 워프가 처리하도록 했다.
5. **빌드 환경 함정(Windows):** CUDA Toolkit은 32비트를 미지원하므로 반드시 "x64 Native Tools" 환경에서 빌드해야 하며(미준수 시 `ACCESS_VIOLATION`), 경로에 한글·공백이 있으면 `cudafe++` 프론트엔드가 비정상 종료한다. 또한 한글 주석의 인코딩(CP949 vs UTF-8) 충돌을 방지해야 한다. 이 함정들은 `README.md`에 별도 경고로 문서화했다.
6. **호스트 측 잔여 순차 작업:** 현재 최댓값 탐색(max_score 찾기)이 GPU 커널이 아니라 호스트에서 O(mn)으로 수행된다(`sw_gpu*.cu`의 `smith_waterman_gpu` 말미). 이는 개선 여지로 6장에서 다룬다.

## 5. 실험 및 성능 평가

### 가. 실험 설정

- **데이터셋:** 티틴(Titin) 단백질. 생쥐(TITIN_MOUSE) **35,213 AA** × 인간(TITIN_HUMAN) **34,350 AA**. 티틴은 알려진 단백질 중 최대 크기로, 점수 테이블이 약 **4.5 GB**에 달하는 실전 규모의 스트레스 테스트다.
- **측정 환경:** NVIDIA RTX 3080 Ti (Ampere), Linux x64. CPU 베이스라인도 동일 머신.
- **측정 지표:**
  - **연산 시간(ms):** CPU는 테이블 채우기 시간, GPU는 H2D+커널+D2H 전체 파이프라인 시간.
  - **GCUPS(Giga Cell Updates Per Second):** 초당 셀 갱신 수 = (m × n) / (시간 × 10⁹). 정렬 하드웨어의 표준 처리량 지표.
  - **Speedup:** CPU 대비 배속.
  - **정확성 검증:** 세 구현의 최대 점수·정렬 길이 일치 여부.
- **비교 대상:** ① `sw_cpu`(CPU 단일 스레드), ② `sw_gpu`(셀 단위 웨이브프론트), ③ `sw_gpu_tiled`(공유 메모리 타일링).

### 나. 실행 시간 비교

실제 벤치마크 출력([99_archive/benchmark_results](99_archive/benchmark_results/))에서 수집한 측정값이다.

| 구현 | 연산 시간 | GCUPS | 최대 점수 | 정렬 길이 | 커널 런치 |
|---|---:|---:|---:|---:|---:|
| `sw_cpu` (CPU 베이스라인) | 66,973 ms | 0.018 | 165,624 | 35,318 AA | — |
| `sw_gpu` (셀 단위 웨이브프론트) | 20,020 ms | 0.060 | 165,624 | 35,318 AA | 69,562회 |
| `sw_gpu_tiled` (공유 메모리 타일) | **14,514 ms** | **0.083** | 165,624 | 35,318 AA | ~2,172회 |

> **정확성 검증 완료:** 세 구현 모두 **최대 정렬 점수 165,624, 정렬 길이 35,318 AA로 완전히 일치**한다. 이는 반대각선 병렬화·타일링·공유 메모리 도입이 알고리즘의 수학적 결과를 손상시키지 않았음을, 즉 병렬화가 **정확하다**는 것을 실증한다. 정확성이 보장되지 않는 가속은 무의미하므로, 이 일치는 본 프로젝트의 가장 중요한 검증 결과다.

### 다. Speedup 및 효율성 분석

#### 1) Speedup

```
CPU 대비 Speedup = T_cpu / T_gpu

sw_gpu       : 66,973 / 20,020 ≈ 3.35×
sw_gpu_tiled : 66,973 / 14,514 ≈ 4.61×
타일 vs 기본 : 20,020 / 14,514 ≈ 1.38×
```

- **셀 단위 GPU만으로도 약 3.4배** 가속. 단순 이식에도 GPU의 대규모 병렬성이 효과를 발휘함을 보여준다.
- **타일링으로 약 4.6배** 까지 향상. 즉 **메모리 최적화가 추가로 1.38배의 가속**을 만들어냈다.

#### 2) 타일링이 1.38배를 만든 원인 분석

1. **커널 런치 오버헤드 제거:** 69,562회 → 2,172회로 **32배 감소**. 각 런치의 고정 비용(런치 지연·스케줄링)이 수만 번 누적되던 것을 제거했다.
2. **전역 메모리 트래픽 감소:** 4.5 GB 테이블은 L2 캐시(수 MB)를 압도하므로 모든 접근이 DRAM에서 일어난다. 셀 단위 버전은 셀마다 의존 셀 3개를 전역에서 재읽기했지만, 타일 버전은 32×32 셀의 모든 중간 계산을 공유 메모리에서 처리하고 전역 접근을 타일 경계 적재/결과 기록으로 최소화했다.
3. **코얼레싱(Coalescing) 개선:** 타일 단위의 정렬된 접근 패턴이 DRAM 메모리 트랜잭션을 합쳐 유효 대역폭을 끌어올렸다.

#### 3) 효율성의 한계 — 왜 GCUPS는 여전히 낮은가

타일 버전의 0.083 GCUPS는 현대 GPU의 이론 처리량(수십~수백 GCUPS)에 한참 못 미친다. 그 원인은 본 문제의 본질적 제약에서 온다.

- **웨이브프론트의 낮은 점유율(Occupancy):** 반대각선 길이가 1 → 최대 → 1로 변하므로, 시작·끝 구간에서는 대부분의 코어가 유휴 상태다. 평균적으로 GPU의 연산 자원이 충분히 채워지지 않는다.
- **메모리 바운드(Memory-bound):** 셀당 연산량(덧셈 3회, 최댓값 3회)은 적은데 4.5 GB 테이블을 한 번씩 읽고 써야 하므로, 연산보다 메모리 전송이 병목이다. 산술 강도(arithmetic intensity)가 낮다.
- **전송 비용(PCIe):** GPU 측정에 4.5 GB 결과 테이블의 D2H 복사가 포함되어, PCIe 대역폭이 무시할 수 없는 비중을 차지한다.

즉 본 결과의 의의는 "절대 GCUPS의 극대화"가 아니라, **동일 알고리즘·동일 하드웨어에서 (1) 병렬화로 3.4배, (2) 메모리 계층 최적화로 추가 1.38배(누적 4.6배)를 정확성 손실 없이 달성**했다는 점, 그리고 **각 최적화 단계의 기여를 정량적으로 분리 측정**했다는 점에 있다.

## 6. 결론

### 가. 결과 요약 및 의의

본 프로젝트는 동적 계획법 기반 Smith-Waterman 지역 정렬 알고리즘을 CUDA로 병렬화하고, 그 효과를 실전 규모(티틴, 4.5 GB 테이블) 벤치마크로 정량 검증했다.

- **알고리즘적 기여:** 셀 단위 O(mn) 순차 의존성을 **반대각선 독립성**으로 분해하여 순차 스텝을 **O(m+n)** 으로 줄이고, 이를 GPU의 SIMT 모델에 매핑했다.
- **시스템적 기여:** 단순 이식(셀 단위, 3.4×)을 넘어, **공유 메모리 타일링·상수 메모리·uint8_t 인코딩**의 메모리 계층 최적화로 **누적 4.6배(타일링 단독 기여 1.38배)** 의 가속을 달성했다. 각 최적화의 효과를 Nsight Compute 프로파일링(uint8 vs int에서 DRAM 읽기 4.6배 차이)과 커널 런치 수(69,562 → 2,172)로 정량 분리했다.
- **정확성 검증:** 세 구현이 **동일한 점수(165,624)·정렬 길이(35,318)** 를 산출하여, 병렬화가 알고리즘의 최적성을 보존함을 실증했다.
- **공학적 완성도:** FASTA 전처리 파이프라인, OS별 빌드 가이드, `CUDA_CHECK` 오류 처리, 사전빌드 바이너리 배포까지 갖춘 재현 가능한 모듈식 시스템을 구축했다.

본 결과의 핵심 교훈은, **GPU 가속은 "병렬화" 한 번이 아니라 "병렬화 + 메모리 계층 설계"의 두 축**으로 이루어지며, 후자의 기여가 결코 작지 않다(이 사례에서 추가 38%)는 점이다.

### 나. 한계 및 추가 최적화 방향

#### 1) 한계

- **O(mn) 메모리:** 트레이스백을 위해 4.5 GB 테이블을 전부 보관한다. 더 큰 서열은 GPU 메모리 한계에 부딪힌다.
- **낮은 처리량(0.083 GCUPS):** 웨이브프론트의 낮은 점유율과 메모리 바운드 특성으로 GPU 이론 성능을 충분히 활용하지 못한다.
- **선형 갭 모델:** 생물학적으로 더 현실적인 **아핀 갭(affine gap, Gotoh)** — 갭 시작과 연장에 다른 벌점 — 을 지원하지 않는다.
- **순차 잔여 작업:** 최댓값 탐색과 트레이스백이 호스트에서 순차로 수행된다.
- **단일 쌍·단일 GPU:** 한 번에 한 쌍만 정렬하며, 데이터베이스 검색 시나리오(1 vs N)나 멀티 GPU는 미지원이다.

#### 2) 추가 최적화 방향

- **메모리 절감 — Hirschberg / 점수-전용 모드:** 점수만 필요하면 직전 두 대각선만 유지해 O(min(m,n)) 공간으로 줄일 수 있고, 트레이스백이 필요하면 분할 정복(Hirschberg)으로 O(mn) 시간·선형 공간을 달성할 수 있다.
- **커널 런치 제거 — CUDA Graphs / Persistent Kernel:** 매크로 웨이브프론트의 2,172회 런치마저 CUDA Graph로 묶거나, 그리드 동기화(cooperative groups) 기반 단일 영속 커널로 통합해 런치 오버헤드를 0에 수렴시킨다.
- **레지스터·워프 셔플:** 타일 내부 의존을 공유 메모리 대신 **워프 셔플(`__shfl`)** 로 처리해 공유 메모리 대역폭과 동기화 비용을 더 줄인다.
- **데이터 타입 다운캐스트 + 포화 연산:** 점수 범위를 분할(블록 단위 재정규화)하여 `int16`/`int8`·DP4A·텐서 코어를 활용, 데이터 밀도와 연산 처리량을 동시에 끌어올린다.
- **아핀 갭(Gotoh) 확장:** E·F 보조 행렬을 추가해 생물학적 현실성을 높인다(메모리 3배 증가는 위 절감 기법으로 상쇄).
- **GPU 측 병렬 리덕션:** 최댓값 탐색을 호스트 O(mn) 대신 GPU 병렬 리덕션으로 옮겨 D2H 전송량(4.5 GB)과 호스트 후처리를 제거한다.
- **다중 정렬·멀티 GPU 배칭:** 데이터베이스 검색용으로 여러 짧은 서열을 인터리빙(inter-sequence parallelism)하여 점유율을 높이고, 여러 GPU에 작업을 분산한다.
